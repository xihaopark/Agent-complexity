# Agentic Complexity Analyzer — 面向 ACL 的课题汇报方案

> **定位**：以发表 ACL 顶级会议为目标，面向导师的体系化、专业化汇报指南  
> **受众**：AI Agent 方向研究生  
> **原则**：轻重结合、创新突出、证据充分、可复现

---

## 一、汇报结构总览（建议时长 25–35 分钟）

| 模块 | 时长 | 核心目标 |
|------|------|----------|

## 二、各模块详细汇报脚本

### 1. 问题与动机（Problem & Motivation）

**汇报要点：**

> 「随着 LLM 驱动的 Agent 系统（如 LangChain、AutoGPT、CrewAI）快速发展，学术界和工业界缺乏一套**可复现、可量化、带证据**的复杂度评估框架。现有工作主要关注：
> - **功能正确性**（如 SWE-bench、HumanEval）
> - **对话质量**（如 MT-Bench）
> - **单次任务成功率**
> 
> 但**架构复杂度、协调开销、LLM 调用成本、可观测性**等维度长期被忽视。本项目提出 Agentic Complexity Analyzer，填补这一空白。」

**关键数据点（可引用）：**

- Agent 系统通常涉及：多 Agent 协作、工具调用链、LLM 推理、状态管理、重试逻辑
- 复杂度失控会导致：调试困难、成本不可控、错误放大、难以维护

**汇报话术示例：**

> 「我们的核心问题是：**给定一个 GitHub 仓库，如何自动、客观地评估其作为 AI Agent 系统的复杂度与可维护性？** 这不仅是工程问题，更是 NLP 系统评估（evaluation of NLP systems）的重要补充。」

---

### 2. 相关工作（Related Work）

**必须覆盖的 ACL/NLP 相关方向：**

| 类别 | 代表工作 | 与本项目的区别 |
|------|----------|----------------|
| LLM 评估 | MT-Bench, Chatbot Arena | 关注对话质量，不关注架构与复杂度 |
| Agent 基准 | SWE-bench, AgentBench | 关注任务完成率，不关注内部结构 |
| 代码复杂度 | Radon, SonarQube | 面向传统软件，无 Agent 特有指标 |
| 软件可观测性 | OpenTelemetry | 运行时追踪，无静态+动态联合分析 |

**汇报话术示例：**

> 「我们将本项目定位在 **NLP 系统评估（System Evaluation）** 与 **AI Agent 工程（Agent Engineering）** 的交集。现有 LLM 评估工作多关注单轮或有限轮对话，而 Agent 系统是长期运行、多组件协作的复杂系统，需要新的评估范式。」

**ACL 定位建议：**

- 主 track：**Resources and Evaluation** 或 **Applications**
- 副 track：可关联 **Generation**（Agent 生成行为）、**Multimodal**（若涉及多模态 Agent）

### 3. 方法原理（Methodology）

#### 3.1 指标体系设计（核心贡献之一）

**汇报逻辑：** 先讲「我们设计了什么」，再讲「为什么这样设计」。

**指标分层结构：**

```
┌─────────────────────────────────────────────────────────────────┐
│  A 系列：架构与静态复杂度（Architecture & Static）                  │
│  A1–A11：模块数、依赖图、调用图、圈复杂度、Halstead、异常、配置      │
│  AG：Agent 模式特征（LLM 客户端、工具注册、Agent 类、记忆模式）       │
├─────────────────────────────────────────────────────────────────┤
│  B 系列：并发与资源（Concurrency）                                 │
│  B1–B3：线程峰值、进程峰值、async 并发度                            │
├─────────────────────────────────────────────────────────────────┤
│  C 系列：协调与通信（Coordination）                                 │
│  C1–C6：Agent 数量、消息频率、带宽、回合数、冗余率、错误放大         │
├─────────────────────────────────────────────────────────────────┤
│  D 系列：LLM 使用（LLM Usage）— 与 ACL 最相关                       │
│  D1–D5：调用次数、Token 用量、Prompt 长度、模板多样性、工具密度      │
├─────────────────────────────────────────────────────────────────┤
│  E 系列：状态与容错（State & Resilience）                           │
│  E1–E2：状态规模、重试/回滚复杂度                                   │
├─────────────────────────────────────────────────────────────────┤
│  F 系列：可测试性与可观测性（Testability & Observability）         │
│  F1–F7：静态/动态可测试性、可观测性、CPU/内存/时延                   │
└─────────────────────────────────────────────────────────────────┘
```

**汇报要点：**

1. **AG 指标**：通过正则 + AST 识别 Agent 特有模式，是**领域适配**的创新。
2. **D 系列**：直接对应 LLM 成本与 Prompt 工程，是 ACL 审稿人最关心的部分。
3. **C 系列**：多 Agent 协调是 Agent 系统独有，传统软件复杂度不覆盖。
4. **阈值设计**：每个指标有 4 档（🟢🟡🟠🔴），基于经验与文献，可解释、可调。

**话术示例：**

> 「我们设计了 33 个指标，覆盖静态、动态、Agent 特有三个维度。其中 **D 系列（LLM 使用）** 与 **C 系列（协调）** 是传统软件复杂度工具所没有的，也是我们面向 Agent 系统的核心创新。」

#### 3.2 静态分析原理

**技术栈：**

- Python：`ast` 解析 + `radon`（圈复杂度、Halstead）+ `networkx`（图分析）
- TypeScript：类似 AST 分析
- 依赖图：import 关系 → 强连通分量（SCC）、模块化（modularity）
- 调用图：函数调用关系 → 入度/中介中心性、圈复杂度

**汇报要点：**

- 依赖图可识别「架构桥梁节点」「循环依赖」
- 调用图可识别「热点函数」「未解析调用比例」
- Agent 模式通过正则匹配：`OpenAI(`, `@tool`, `class *Agent`, `VectorStore(` 等

#### 3.3 动态分析原理（创新重点）

**流程：**

1. **代码插桩**：用 `libcst` 在 Python 源码中注入 `ac_track_call(kind, target, func, *args, **kwargs)`
2. **调用分类**：根据 `target` 名称分类为 `llm_call` / `tool_call` / `message_call` / `retry_call`
3. **运行时追踪**：`ac_track_call` 内部记录 `call_start` / `call_end` / `call_error`，写入 JSONL
4. **指标计算**：解析 JSONL，统计 D/C/E 系列指标

**汇报要点：**

- **非侵入式**：通过 AST 变换注入，不修改业务逻辑
- **可扩展**：新增 `kind` 只需改 `_classify()` 和指标计算
- **Token 提取**：从 LLM 响应中解析 `usage` 字段，支持 OpenAI/Anthropic 等格式

#### 3.4 沙箱与可复现性

- Docker 容器隔离，资源限制（CPU、内存、进程数）
- 入口命令自动检测：`.agent-complexity.yml` > `main.py` > `pyproject.toml` > README
- 多轮运行（repeats） + Bootstrap 置信区间，增强统计可靠性

### 4. 系统架构与实现（Architecture & Implementation）

#### 4.1 整体架构图（建议画图）

```
┌──────────────┐     POST /analyses      ┌──────────────┐
│   Web (Next) │ ──────────────────────▶ │  API (FastAPI)│

┌──────────────┐     Celery Task         ┌──────────────┐
│  PostgreSQL  │ ◀────────────────────── │   Worker     │
│  Redis       │     status/metrics      │  (sandbox)   │
└──────────────┘                         └──────┬──────┘
                                                 │
                    ┌────────────────────────────┼────────────────────────────┐
                    │                            │                            │
                    ▼                            ▼                            ▼
             clone repo              static analysis              instrument + run
             (git)                   (AST, radon, nx)            (Docker, events)
```

#### 4.2 软件工程难点与解决方案

| 难点 | 解决方案 | 汇报话术 |
|------|----------|----------|
| **私有仓库 Token 安全** | Token 存 Redis，短期 TTL，不落库 | 「我们采用短期存储 + 任务完成后立即清除，符合安全最佳实践。」 |
| **Docker-in-Docker** | Worker 挂载宿主机 `docker.sock` | 「分析在隔离容器内执行，同时复用宿主机 Docker，避免嵌套带来的性能损耗。」 |
| **入口命令未知** | 多级 fallback 检测 | 「支持声明式配置、启发式检测、README 解析，覆盖主流 Python/Node 项目。」 |
| **异步调用追踪** | Patch `asyncio.create_task` + 引用计数 | 「正确统计 async 并发度，需 hook 事件循环，我们实现了透明追踪。」 |
| **指标可解释性** | 每指标绑定 `evidence_ref` | 「所有指标可追溯到 JSON/Parquet 证据文件，支持审稿人复现。」 |

### 5. 创新点与贡献（Contributions）

**建议按「可发表」角度提炼：**

1. **指标体系**：首个面向 LLM Agent 系统的**静态+动态联合**复杂度指标体系，含 33 个可解释指标。
2. **Agent 模式识别**：通过 AST + 正则实现 **AG** 指标，自动识别 LLM 客户端、工具注册、Agent 类等。
3. **透明插桩**：基于 libcst 的**非侵入式**运行时追踪，支持 llm_call / tool_call / message_call 分类。
4. **证据链设计**：每指标绑定 `evidence_ref`，报告可复核，满足学术可复现要求。
5. **端到端平台**：从 GitHub URL 到结构化报告的**一键分析**，开源可复现。

**汇报话术：**

> 「我们的贡献可以概括为：**一套指标 + 一套工具 + 一个平台**。指标设计考虑了 Agent 系统的特殊性；工具实现了静态与动态的联合分析；平台降低了使用门槛，便于社区采用和扩展。」

### 6. 实验与评估（Experiments）

**当前状态：** 项目有单元测试，但尚未有大规模实验。为 ACL 投稿，建议补充：

1. **基准仓库集**：选取 20–50 个公开 Agent 项目（LangChain、LangGraph、CrewAI 等生态）
2. **指标分布分析**：各指标在基准集上的分布、相关性
3. **案例研究**：选 2–3 个项目做深度分析，展示报告价值
4. **人工评估**：邀请开发者对报告有用性打分（1–5），做简单用户研究

**汇报话术（若实验未完成）：**

> 「我们已完成系统实现与单元测试。下一步计划在 X 个开源 Agent 项目上开展基准实验，并邀请 Y 位开发者进行用户研究，以验证指标的有效性与报告的实用性。」

### 7. 局限与未来工作（Limitations & Future Work）

**建议坦诚提及：**

1. **语言覆盖**：目前主要支持 Python，TypeScript 分析较浅
2. **入口检测**：复杂项目可能检测失败，需用户手动指定
3. **指标权重**：综合得分 S 目前为简单平均，可引入学习到的权重
4. **多模态**：未覆盖视觉/语音 Agent 的复杂度

**未来工作方向：**

- 扩展到更多语言（如 Go、Rust）
- 与 SWE-bench 等基准结合，探索复杂度与任务成功率的关系
- 引入 LLM 对报告进行摘要与建议生成

## 四、常见问题准备（Q&A）

| 问题 | 建议回答 |
|------|----------|
| 和 SonarQube 有什么区别？ | SonarQube 面向传统软件，无 Agent 指标（D/C/AG），无动态 LLM 追踪。 |
| 为什么选这些阈值？ | 基于文献与经验，部分参考 Radon/Sonar 惯例，可配置、可调。 |
| 插桩会不会影响行为？ | 仅增加日志写入，不改变控制流；且分析在隔离沙箱中，不影响生产。 |
| ACL 更关注什么？ | 建议强调 D 系列（LLM 使用）、C 系列（协调）、可复现性与证据链。 |
| 如何证明指标有效？ | 计划通过基准实验 + 用户研究；也可与任务成功率做相关性分析。 |
