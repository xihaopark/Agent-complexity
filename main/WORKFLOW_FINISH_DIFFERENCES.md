# 根目录 workflow 与 -finish workflow 差异说明

## 说明目的

本文档用于解释本仓库中两类 workflow 组织方式的本质差异：

- 根目录下的原始 workflow
- `-finish` 目录下的 finish workflow

这两类 workflow 并不是简单的“同一套文件放在不同目录”，而是面向不同目标的两种工作流表达方式。

## 一句话结论

- 根目录 workflow 是“原始全流程定义”
- `-finish` workflow 是“面向分步完成与评估的重组版本”

如果用更直接的话来说：

- 根目录回答的是“这个分析从头到尾应该怎样做”
- `-finish` 回答的是“如果把这个分析拆成多个明确阶段，每一阶段如何独立完成并证明已完成”

## 最本质的两个区别

### 1. 顺序约束来源不同

根目录 workflow 和 `-finish` workflow 都不是“可以随意换顺序”的。

真正的区别不在于“一个固定、一个随意”，而在于：

- 根目录 workflow 的顺序主要由协议层主导
- `-finish` workflow 的顺序主要由依赖层主导

根目录下除了 Snakemake 自身的依赖关系之外，还额外叠加了一层 `agent-steps.md` 的高层执行协议。

这层协议通常明确要求：

- 按编号顺序执行
- 当前步骤未通过验证前不能进入下一步
- 失败后停止，而不是跳过或重排后续步骤

因此，对 agent/LLM 来说，根目录 workflow 的执行顺序并不是根据“我当前只想要哪个输出”来自由调整，而是先被协议层写死，再在协议允许范围内调用原始 workflow。

换句话说，根目录 workflow 的特点不是单纯“有 DAG”，而是“有 DAG 之外，还存在一层显式的阶段顺序协议”。

`-finish` workflow 则没有同等级的全局 `agent-steps.md` 协议来规定必须按 1、2、3、4 的编号顺序推进。

它的特点是：

- 整体流程被拆成多个 step 文件
- 每个 step 都是一个独立的完成单元
- 执行者可以显式选择调用哪个 step
- 但是否能成功执行，仍然取决于前置输入和中间产物是否已经存在

因此，`-finish` workflow 的顺序不是“完全自由”，而是“没有被高层协议写死，但仍受真实数据依赖约束”。

最简洁地说：

- 根目录 workflow 是“协议主导顺序”
- `-finish` workflow 是“依赖主导顺序”

### 2. 交付目标不同

根目录 workflow 的目标是忠实表达原始生信分析流程本身。

它更关注：

- 分析逻辑是否完整
- 依赖关系是否正确
- 最终结果是否能从原始输入自然推导出来

因此，根目录 workflow 更像“研究流程定义层”。

`-finish` workflow 的目标则不是单纯让流程能跑通，而是让流程变成：

- 可分阶段交付
- 可逐步检查
- 可逐步复现
- 可被 agent 或外部系统分步骤完成

因此，`-finish` workflow 更像“任务拆分与完成态交付层”。

## 为什么目录结构会不同

这两个本质差异会直接体现在目录结构上。

### 根目录 workflow 的目录倾向

根目录 workflow 通常更强调：

- 主 `Snakefile`
- `workflow/rules/` 下的模块化规则
- 与原始工作流一致的结构组织
- 在 workflow 内部表达完整依赖逻辑

它更像一个标准的 Snakemake 工作流仓库结构。

### -finish workflow 的目录倾向

`-finish` workflow 更强调：

- `steps/` 中逐步拆开的 step 文件
- `config_basic/` 中本地化配置
- 本地化测试输入目录，如 `ngs-test-data/`
- 本地化资源目录，如 `resources/`
- 独立的 `logs/` 和 `results/`

这种结构的重点不是“尽量少文件”，而是“让每一个阶段都更容易被单独调用、单独验证、单独交付”。

## 为什么 -finish 里会本地化输入和资源

根目录 workflow 可以依赖原始仓库自身的测试目录、配置目录和资源路径。

但 `-finish` workflow 为了成为更独立的“完成单元”，通常会把以下内容本地化到自己的目录中：

- 配置文件
- 测试输入
- 参考资源
- 日志目录
- 结果目录

这样做有几个作用：

- 降低对原始目录结构的耦合
- 让每个 finish workflow 更接近自包含
- 方便单步执行和单步检查
- 便于比较不同 workflow 在相同 finish 框架下的处理方式

## 从任务视角理解两者差异

如果从“使用者到底在做什么”这个角度看，两者区别更清楚。

### 使用根目录 workflow 时

使用者更像是在遵循一份已经写好步骤顺序的执行合同，去完成一个完整分析流程。

他的关注点通常是：

- 当前阶段是否按协议完成
- 当前阶段是否通过验证
- 最终产物是否在完整流程结束后生成

换句话说，使用者关心的是“流程是否按既定协议完整完成”。

### 使用 -finish workflow 时

使用者更像是在面对一组已经拆开的 step，并根据依赖情况逐步推进任务。

他的关注点通常是：

- 当前 step 是否完成
- 当前 step 产物是否正确
- 下一步所需依赖是否已经满足

换句话说，使用者关心的是“每一个阶段能否在依赖满足的前提下独立完成与验收”。

## 从 agent 评估视角理解两者差异

本仓库的重要用途之一，是把 workflow 拆成多个步骤，用于测试大模型或 agent 对各步骤的完成能力。

在这个背景下：

- 根目录 workflow 更接近原始问题本身
- `-finish` workflow 更接近评测或执行任务本身

也就是说：

- 根目录提供的是“完整问题”
- `-finish` 提供的是“可逐步求解的问题分解形式”

因此，`-finish` 目录可以看作是面向任务执行与评估的重新组织版本，而不是单纯的备份目录。

## 可以向导师如何概括

可以用下面这段话概括：

根目录中的 workflow 属于原始工作流定义层，但在本仓库的 agent 执行场景下，又额外叠加了一层 `agent-steps.md` 规定的固定高层顺序，因此它更接近“协议主导的完整流程执行”；`-finish` 目录中的 workflow 属于任务拆分与完成态交付层，它把流程拆成多个可独立调用的步骤，不再由同等级的全局协议统一写死顺序，而是主要由步骤之间的真实依赖关系决定执行先后。因此，两者最本质的区别不是一个能否换顺序，而是一个由协议主导顺序，一个由依赖主导顺序。

## 最终总结

根目录 workflow 和 `-finish` workflow 的差异，不是“文件多一点还是少一点”，也不是“路径写法不同”这么表层的问题。

最本质的区别只有两点：

- 根目录 workflow 以协议主导的完整流程执行为中心
- `-finish` workflow 以依赖主导的分步完成为中心

由此进一步带来的结构差异包括：

- 根目录强调统一入口、全局依赖以及额外的高层顺序协议
- `-finish` 强调 step 拆分、外部调用以及依赖满足后的逐步推进
- 根目录更像原始分析工作流
- `-finish` 更像可交付、可评估、可逐步完成的任务工作流

这就是两者之间最核心、也最适合向导师解释的区别。
