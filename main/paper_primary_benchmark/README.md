# Paper primary benchmark（论文主实验集）

本目录是 **MAIN_WORKFLOW_SET（v2）** 的固定「项目视图」：主实验 **30** 个 workflow + **3** 个 smoke，作为后续 baseline / 对比 / 复跑的**默认对象**。  
权威说明与入选理由见：`main/refine-logs/MAIN_WORKFLOW_SET.md`。

## 目录结构

| 路径 | 含义 |
|------|------|
| `manifest.json` | 机器可读清单（id、相对 `main/` 的路径、steps、family、kind） |
| `workflows/` | 指向 `main/finish/<workflow_id>/` 的**符号链接**（主实验 30 个） |
| `smoke_workflows/` | 同上，**3** 个 smoke（先跑通再跑 30） |
| `task_definitions/` | **Pipeline task 覆盖层**（每主实验 workflow 一个 JSON）：将 finish `manifest.json` 中的 **steps** 切分为若干 **task**，含元数据、`inputs`/`outputs`、`step_snapshots`（可从 manifest 复制粘贴维护） |
| `task_definitions/schema.json` | 上述 JSON 的 JSON Schema（可选校验） |
| `tools/validate_task_definitions.py` | 校验 task 划分与 `finish/.../manifest.json` 中 **steps 顺序**一致（完整划分、无重复） |
| `tools/generate_task_definitions.py` | 按 manifest 顺序**自动切分**为连续 stage（默认约每 4 步一个 task）；**不覆盖**已有文件，除非 `--force` |

Workflow 资产不复制：所有真实文件仍在 `main/finish/`；此处仅占位与统一入口。**不在 `main/finish/` 内写入 task 定义**，避免污染既有 manifest / runner。

### Pipeline task 与 trace「stage」的区别

- **Pipeline task（本目录 `task_definitions/`）**：面向论文与实验管理，把 Snakemake finish **step** 粗粒度合并为「数据预处理 / 定量 / 差异分析」等阶段；执行真源仍是各 workflow 的 `manifest.json`。
- **Trace stage（`main/finish/R_WORKFLOW_TRACE_PROTOCOL.md`）**：描述 **agent 回合**（如输入是否就绪、选步、执行、验证），与 pipeline 分段不是同一概念；命名上勿混用。

## 实验脚本怎么用

- 遍历主集：读 `manifest.json` 的 `main_workflows`，或用 `workflows/*` 解析到的路径（解析后为 `../finish/<id>`）。
- 仅跑 smoke：`smoke_workflows` 或 `manifest.json` → `smoke_workflows`。

从仓库根目录举例（路径按你的 cwd 调整）：

```bash
# 主实验 30 个根目录（经 symlink）
ls -1 main/paper_primary_benchmark/workflows

# 校验全部 task 定义（仓库根目录执行）
python3 main/paper_primary_benchmark/tools/validate_task_definitions.py --main-root main

# 仅校验单个 workflow
python3 main/paper_primary_benchmark/tools/validate_task_definitions.py --main-root main --workflow-id rna-seq-kallisto-sleuth-finish

# 为新加入主集的 workflow 生成默认 task 划分（不覆盖已有手写 JSON）
python3 main/paper_primary_benchmark/tools/generate_task_definitions.py --main-root main
```

## 与 refine-logs 的关系

- 集合成员以 `main/refine-logs/MAIN_WORKFLOW_SET.json` 为准；若日后增删成员，应**先改**该 JSON（并同步 `MAIN_WORKFLOW_SET.md`），再更新本目录下的 symlink 与 `manifest.json`（避免清单与链接漂移）。

## 更新清单与链接（维护者）

在修改 `MAIN_WORKFLOW_SET.json` 后，需要：

1. 为新 id 在 `main/finish/<id>/` 中存在；
2. 在 `workflows/`（及如需要的 `smoke_workflows/`）下添加 `ln -s ../../finish/<id> <id>`；
3. 更新 `manifest.json` 中对应条目（可从 `MAIN_WORKFLOW_SET.md` 表格抄 steps / family / kind）。
4. 为该 workflow 新增或生成 `task_definitions/<workflow_id>.json`（可先生成默认划分再手工改 `pipeline_stage_label` / I/O）。

## Task 定义维护说明

- **参考示例**：`task_definitions/rna-seq-kallisto-sleuth-finish.json`、`task_definitions/epigen-fetch_ngs-finish.json` 为手写分段；其余 28 个主实验 workflow 为 **自动生成** 的连续分段（`metadata.autogenerated: true`），可按论文需要逐步改为语义化 stage 名并补全 I/O。
- **重新生成默认文件**（会覆盖）：`python3 main/paper_primary_benchmark/tools/generate_task_definitions.py --main-root main --workflow-id <id> --force`

## ldp R-task 评测（BixBench 式 rollout，无 Snakemake）

面向 **以 R 为主的独立 task**：agent 在预置工作目录内通过 **shell / Rscript / 读写文件** 完成目标，**不** 经过 Snakemake 或历史 `finish` runner。实现位于 `ldp_r_task_eval/`，与上方「pipeline task / manifest」正交；需要跑主集 workflow 时仍用 `manifest.json` 与 `task_definitions/`，需要 **ldp + SimpleAgent** 的 R 沙箱评测时用本节。

| 路径 | 含义 |
|------|------|
| `ldp_r_task_eval/r_task_env.py` | `RTaskEvalEnv`：`reset` / `step` 与 `ldp` Agent 对接 |
| `ldp_r_task_eval/rollout.py` | `vanilla_r_task_rollout`、`save_run_artifacts`（`trajectory.jsonl` + `metadata.json`） |
| `ldp_r_task_eval/run_pilot.py` | 单 task pilot；`--smoke` 为无 LLM 的脚本化成功路径 |
| `ldp_r_task_eval/batch_runner.py` | 读取 `r_tasks/registry.json` 批量跑（可与 `manifest.json` 并行维护） |
| `ldp_r_task_eval/r_tasks/registry.json` | **146** 条：`pilot_hello`（`ready`）+ **145** 个 pipeline stage 占位（`stub`） |
| `ldp_r_task_eval/tasks/stubs/<workflow_id>__<task_id>/` | 与 `task_definitions` 一一对应的评测工作区占位（`OBJECTIVE.md`、`meta.json`、`input/`、`output/`） |
| `ldp_r_task_eval/tools/generate_r_task_stubs.py` | 从 `task_definitions/*.json` 生成/更新占位与 `registry.json`（`--dry-run` / `--force`） |
| `ldp_r_task_eval/tools/validate_r_task_registry.py` | 校验注册表条目数与磁盘目录、`meta.json` 是否一致 |
| `ldp_r_task_eval/tasks/pilot_hello/` | 示例 workspace（`OBJECTIVE.md` + `input/numbers.txt`） |
| `ldp_r_task_eval/runs/` | 默认轨迹输出根目录 |
| `ldp_r_task_eval/config/pilot_example.yaml` | `max_steps`、agent `llm_model` / `sys_prompt`、task 路径 |
| `ldp_r_task_eval/config/pilot_openrouter.yaml` | 同上，但 `llm_model.name` 为 LiteLLM 的 `openrouter/...` 模型 id |
| `ldp_r_task_eval/llm_env.py` | 从文件注入 `OPENROUTER_API_KEY`（供 LiteLLM 走 OpenRouter） |
| `ldp_r_task_eval/CONTRACTS.md` | `DataAnalysisEnv` / `Agent` 异步契约摘要与独立 venv 命令 |

**依赖**：Python ≥3.12 推荐；`pip install -r main/paper_primary_benchmark/ldp_r_task_eval/requirements.txt`。

**Agent**：使用 `ldp` 自带的 **`SimpleAgent`**（见 `ldp/agent/simple_agent.py`）：`init_state` 保存工具列表；每步 **`get_asv`** 里把当前对话（含 user / tool 消息）交给 **`LLMCallOp`**（底层为 **LiteLLM**），按 `llm_model` 字典调用模型并解析出 **`ToolRequestMessage`**（工具调用），再交给 `RTaskEvalEnv.step`。`sys_prompt` 会经 `prepend_sys` 拼进消息。无需单独写 ReAct 循环，**`vanilla_r_task_rollout`** 已按 BixBench 式 observe→act 接好。

**OpenRouter**：在仓库根放一行 key 的 `openrouterkey.txt`（已在 `.gitignore`），或自行 `export OPENROUTER_API_KEY=...`。非 `--smoke` 时 `run_pilot` / `batch_runner` 默认读取该文件并设置环境变量；LiteLLM 使用 `openrouter/<provider>/<model>` 形式的 `agent.llm_model.name`（见 `pilot_openrouter.yaml`）。

**无 API 的冒烟**（写入 `output/result.txt` 并 `submit_done`，落盘 `ldp_r_task_eval/runs/<run_id>/`）：

```bash
python3 main/paper_primary_benchmark/ldp_r_task_eval/run_pilot.py \
  --smoke \
  --config main/paper_primary_benchmark/ldp_r_task_eval/config/pilot_example.yaml \
  --run-id my_smoke
```

**接入真实模型**（去掉 `--smoke`；OpenRouter 示例配置如下，key 默认从仓库根 `openrouterkey.txt` 读取）：

```bash
python3 main/paper_primary_benchmark/ldp_r_task_eval/run_pilot.py \
  --config main/paper_primary_benchmark/ldp_r_task_eval/config/pilot_openrouter.yaml \
  --run-id pilot_openrouter
```

**145 个 pipeline stage 占位**（与 Snakemake 分段元数据对齐；**不含**真实测序数据或金标准，需后续人工补全）：

```bash
# 生成或更新 stubs + registry.json（首次会创建 145 个目录）
python3 main/paper_primary_benchmark/ldp_r_task_eval/tools/generate_r_task_stubs.py

# 慎用 --force：会覆盖已有 stub 内的 OBJECTIVE.md / meta.json。
# 若已跑过 sample_50 或手工补过任务，--force 会破坏微型评测内容；应先备份或仅对未准备目录使用。

# 校验：应为 146 任务（145 stub + 1 pilot）
python3 main/paper_primary_benchmark/ldp_r_task_eval/tools/validate_r_task_registry.py
```

**stub → ready（可评测）**：在对应 `tasks/stubs/.../` 下补 `input/` 数据、写清 `OBJECTIVE.md` 验收指标；在 `r_tasks/registry.json` 中将该条目的 **`status`** 改为 **`ready`**（并确认 `success_artifact_glob` 与 `submit_done` 策略一致）。勿向 Git 提交大体积数据（LFS 或外链）。

**Sample 50（按比例 + 5×10 agent 分批）**：从 145 个 stub 按 **family** 比例抽 **50** 条，每条已备齐 **`input/values.txt`**、**`OBJECTIVE.md`**、**`evaluation/reference_sum.txt`**（微型求和任务，与 `pilot_hello` 同构，不捆绑真实 FASTQ）。主注册表中的对应行已标 **`ready`** 与 **`agent_batch`**；独立 50 条注册表见 `r_tasks/registry.sample_50.json`，说明见 `ldp_r_task_eval/experiments/sample_50/README.md`。

**评测语义**：`RTaskEvalEnv` 的轨迹奖励只要求 **`output/result.txt` 存在**；数值是否正确需用 `evaluation/reference_sum.txt` 或自写脚本离线比对。`--smoke` 路径已改为按工作区自动写入**与参考答案一致**的内容（见 `rollout.expected_result_text_for_smoke`），避免 smoke 与微型任务答案脱节。

```bash
# 重新抽样并写文件（默认 seed=42；会 patch 主 registry.json）
python3 main/paper_primary_benchmark/ldp_r_task_eval/tools/sample_and_prepare_eval_50.py --write-registry

python3 main/paper_primary_benchmark/ldp_r_task_eval/tools/validate_r_task_registry.py --registry main/paper_primary_benchmark/ldp_r_task_eval/r_tasks/registry.sample_50.json --kind sample50

# 仅对这 50 条跑 batch smoke
python3 main/paper_primary_benchmark/ldp_r_task_eval/batch_runner.py \
  --registry main/paper_primary_benchmark/ldp_r_task_eval/r_tasks/registry.sample_50.json --smoke
```

**批量**（registry 默认 `ldp_r_task_eval/r_tasks/registry.json`）：单条失败会记录并继续；退出码 **0** 为全部成功，**1** 为至少一条失败。

```bash
python3 main/paper_primary_benchmark/ldp_r_task_eval/batch_runner.py --smoke
```

注：根目录 `ldp_r_task_eval/r_tasks.json` 已弃用，请只用 `r_tasks/registry.json`。
