## Key observations

- **Does PAV beat vanilla?** No-skill: PAV 2 vs vanilla 2. With-skill: PAV 1 vs vanilla 1. â†’ PAV is neutral on pass count overall.
- **Skill on `akinyi_deseq2` (only task with a paper skill)**: vanilla_no_skill=pass, vanilla_with_skill=pass, pav_no_skill=pass, pav_with_skill=pass.
- **Tasks no LLM arm passes (10)**: `riya_limma`, `snakepipes_merge_ct`, `chipseq_plot_peak_intersect`, `chipseq_plot_macs_qc`, `chipseq_plot_homer_annot`, `snakepipes_scrna_merge_coutt`, `snakepipes_scrna_qc`, `epigenbutton_mapping_stats`, `epigenbutton_peak_stats`, `smartseqtotal_violin`. These define the current hardness ceiling for GPT-4o on this benchmark.
- **Tasks where injecting the skill flipped a failure to a pass**: none.
- **Tasks where PAV (no skill) flipped a vanilla failure to a pass**: none.
- **Tasks where *adding* skill injection regressed a pass to a non-pass**: `snakepipes_merge_fc` (vanilla **pass â†’ partial** when skill manifest is supplied; same with PAV). For this task the manifest provides the no-skill marker (no paper is mapped), so the regression must be attributed to per-run stochasticity, not to skill content â€” a single-roll sample is noisy.

## Recommendations for the next iteration

1. **Expand paper-skill coverage**. Today only 1/12 tasks has a skill
   (`akinyi_deseq2`); on that task all four arms already pass because
   the shipped R script is a perfect recipe. Add skills for
   `snakemake-workflows-chipseq` (ENCODE guidelines / MACS2),
   `RiyaDua-cervical-cancer-snakemake-workflow` (limma, Ritchie 2015),
   `maxplanck-ie-snakepipes-finish` (SnakePipes 2019), and
   `gersteinlab-ASTRO` (SmartSeq-Total) to make the ablation discriminative.
2. **Re-target `akinyi_deseq2`'s mapped DOI to Love 2014 (DESeq2)**.
   The current Conesa 2016 survey is generic; the Love 2014 skill already
   has runnable `DESeqDataSetFromMatrix â†’ DESeq â†’ results` snippets
   (see Subagent D's _STATUS Â§Recommendations #2). This would also give
   `akinyi_deseq2` a stronger skill-vs-no-skill signal.
3. **Harden the numeric tolerance / PDF handling in the evaluator**.
   Many chipseq/scrna tasks are currently *partial* because the output
   file exists but differs byte-wise (PDF plots are never
   byte-identical across R sessions; TSV cell formatting differs).
   Consider (a) a per-row diff-tolerant comparator for TSVs and
   (b) a PDF *visual similarity* or *data-extraction* check so these
   tasks can become passable.
4. **Add harder tasks**. The current benchmark has only 1 `hard` task
   (`snakepipes_scrna_qc`, 6 expected files). All other tasks are easy/
   medium and the LLM converges quickly thanks to the shipped R script
   in the workspace. Introducing tasks where the R script is **not**
   shipped (or is deliberately broken) would unlock the PAV-vs-vanilla
   differential signal â€” right now both agents effectively just run the
   provided script.
5. **Repeat the sweep with `seed`/repetition**. All 48 runs here are
   single-rolls. With partial/fail noise at 8-17% pass-rate, a 12-task
   benchmark can't reliably distinguish Â±1 task differences. Run each
   (arm, task) â‰Ą 3Ă— and report mean Â± sd.
6. **Ablate script-in-workspace**. Run the same sweep with the canonical
   `*.R` script *removed* from `workspace/` â€” this is the scenario the
   skill is *actually designed for* and would likely make the
   skill-vs-no-skill gap appear.
