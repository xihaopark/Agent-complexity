## Spot-check: did the agent's plan reflect the injected skill?

Injection itself is confirmed by the audit (sha256 + char_len match the
Phase-2 resolver for all 18 non-`none` runs, and the `none` runs carry
`arm: "none"` + `injected: false` with the sentinel string). The SimpleAgent
does not retain the system prompt inside `trajectory.jsonl::agent_state.messages`
(it is built on-the-fly per LLM call from `config.sys_prompt` with `{{SKILL_MD}}`
substituted by the batch runner), so we sanity-check by inspecting the first
tool call (`write_plan`) emitted by the agent per arm.

### `akinyi_deseq2` (all 4 arms = pass)

- **`none`**: plan is `"1. Read the featureCounts-style count matrix ...
  2. Filter out rows starting with 'ERCC-' ... 3. Prepare the count matrix
  for DESeq2 analysis ... 4. Run DESeq2 differential expression ..."` —
  canonical DESeq2 recipe, clearly from the shipped `run_deseq2_analysis.R`
  + `OBJECTIVE.md` that remain in the workspace.
- **`paper`** (Conesa 2016 survey): plan reads `"... 3. Prepare the data
  for DESeq2 analysis by creating a DESeqDataSet object. 4. Run DESeq2
  differential expression analysis with the design formula ..."` — explicit
  mention of `DESeqDataSet` (a named Bioconductor construct prominent in
  the Conesa 2016 "Parameters / Notes for R-analysis agent" section).
- **`pipeline`**: plan reads `"2. Load the DESeq2 library in R ...
  4. Set up the DESeq2 dataset using the count matrix and condition
  information ..."` — speaks in pipeline-convention terms ("load library",
  "set up dataset"), matching B2's workflow-structure skill.
- **`llm_plan`**: plan reads `"1. Load the necessary R libraries,
  specifically DESeq2. 2. Read the input count matrix ... 3. Remove rows
  where the Geneid starts with 'ERCC-' ..."` — faithful echo of the
  OBJECTIVE.md steps (which is exactly what the llm_plan skill is
  grounded on).

All 4 arms converge on the same pass output because the shipped
`run_deseq2_analysis.R` remains on disk (see Recommendation #1). The
differential signal between arms is behavioural (vocabulary) not
terminal (verdict) here.

### `methylkit_unite` (paper=fail, pipeline=fail, llm_plan=fail, none=partial)

- **`none`**: `"1. Load the necessary libraries in R, specifically
  methylKit. 2. Read the bismark coverage files using methylKit::methRead
  with the specified parameters. 3. Unite the methylation data using
  methylKit::unite with the given parameters. 4. Save the united
  methylBase object as mk_united.rds ..."` — OBJECTIVE-grounded.
- **`paper`** (`7a926c67…` = fritjoflammers methylanalysis review skill):
  `"1. Load the methylKit library in R. 2. Use methylKit::methRead to
  read the four bismark-style coverage files with the specified
  parameters. 3. Run methylKit::unite on the loaded data. 4. Save the
  united methylBase object using saveRDS ..."` — mentions the four
  samples explicitly (a detail in the paper skill).
- **`pipeline`** (`98666787…` = fritjoflammers methylanalysis
  pipeline-convention skill): `"1. Load the four per-sample bismark-style
  coverage files using methylKit::methRead with the specified
  parameters. 2. Use methylKit::unite to unite the methylation data.
  3. Save the united methylBase object to output/mk_united.rds using
  saveRDS ..."` — pipeline-style terse stepwise, no methodological
  preamble.
- **`llm_plan`** (`dc8aedc7…`): `"1. Load the necessary R package
  methylKit. 2. Define the file paths ... 3. Use methRead to read the
  coverage files with the specified parameters: pipeline='bismarkCoverage',
  mincov=4, treatment=c(0, 0, 1, 1), and assembly='mock_v1'. 4. Use
  unite ..."` — the most verbose of the four, listing every
  `methRead` keyword verbatim. This is consistent with the llm_plan
  SKILL having been generated directly from OBJECTIVE.md.

The fact that all four arms produce a correct-looking methylkit plan
yet three verdict-fail (and one is partial) suggests the methylkit
tasks are limited by evaluator tolerance / byte-identity rather than by
the agent's semantic understanding — a case for Recommendation #2
(per-row tolerance evaluator).

## Recommendations for the next iteration

1. **Remove shipped R scripts from canonical task workspaces.** As long as `run_*.R` sits next to `OBJECTIVE.md`, the agent can ignore the skill entirely; the current `none` arm still passes 2/6 largely because of that recipe-in-workspace shortcut. Hiding those files (or substituting a deliberately-broken one) is the fastest way to make the skill source *actually* decisive.

2. **Per-row tolerance evaluator for methylation outputs.** `methylkit_unite` and `methylkit_to_tibble` hover between partial and fail across arms — the generated tables exist but don't match byte-for-byte (row order / float formatting). Ship the row-diff tolerant comparator mentioned in Subagent D's status so these rollouts can resolve as pass/fail on content.

3. **Expand to more tasks per family.** 3 rna vs 3 methylation is enough to show arm differences but every per-family cell has only 3 samples. Adding 2–3 more ChIP-seq / scRNA tasks (with paper+pipeline+llm_plan skills generated in advance) would let us make per-family claims at ≥5 samples/cell.

4. **Layer PAV back on top of the winning skill source.** The V1 PAV runs were inconclusive because the skill coverage was 1/12; with the Phase-3 winner (paper) giving full coverage, re-running PAV with that skill source isolates the planning-discipline lift on top of a strong recipe.

5. **Seed / repeat the sweep.** Still 6 single-roll rollouts per arm; verdict deltas of ≤1 task are noise at this N. A 3×-repeat with the same TS prefix would give mean±sd on pass_rate.
