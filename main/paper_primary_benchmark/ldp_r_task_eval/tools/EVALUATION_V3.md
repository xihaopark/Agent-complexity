# Evaluation V3 — insight-oriented scoring rubric

V3 is a **strictly additive** layer on top of V2. The V2 evaluator
(`tools/evaluate_real_run_v2.py`) and V1 evaluator
(`tools/evaluate_real_run.py`) are **not modified**. V3 reads V2's output
— either by calling V2 in-process or by loading cached `<batch>.v2.json`
— and appends a deterministic **insight block** that explains *why* a
given task passed or failed.

No LLM calls, no network, no new dependencies beyond what V2 already
ships with.

## Insight block per task

For every task record V3 adds a field `insight`:

```jsonc
{
  "failure_mode": "row_drift",              // enum; see catalog below
  "confidence": "medium",                   // high / medium / low
  "actionable_fix": "cells matched 842/1000 (84%) on `counts.tsv`",
  "arm": "pipeline",                        // paper / none / pipeline / llm_plan
  "skill_tokens_matched": ["DESeq2", "DESeqDataSetFromMatrix"],
  "skill_tokens_total": 12,
  "skill_tokens_coverage": 0.167,
  "steps_total": 11,
  "writes_after_first": 4,
  "rscript_errs": ["Error in if(...) : ..."],
  "per_file_diff": [
    {"filename": "counts.tsv", "mode": "row_drift",
     "note": "schema-aligned, cells 84%, rows 12% (score 0.63)"}
  ]
}
```

The per-task `overall_score` / `verdict` from V2 are preserved verbatim;
V3 only adds qualitative, diagnostic context.

## Failure-mode catalog

| mode | trigger | example fix |
|------|---------|-------------|
| `ok` | V2 verdict is `pass`, or all per-file modes are `ok` | — |
| `no_rscript_call` | no R exit-0 observation AND ≤ 2 meaningful tool calls AND no output produced | "agent never invoked R; prompt may not have been loaded into context" |
| `rscript_crashed` | ≥ 1 R error in trajectory, no valid output file | "R error: Error in if(!file.exists(location)) ..." |
| `output_missing` | R ran, but an expected file is absent | "rscript exited 0 but expected file(s) absent: [...]" |
| `schema_drift` | file present, column count differs from reference (or column Jaccard < 0.5) | "produced K cols vs reference K' (shared S) on `x.tsv`" |
| `row_drift` | file present, columns match, but cells_matched < 0.8 · total_cells | "cells matched 842/1000 (84%) on `x.tsv`" |
| `float_drift` | schema match, ≥ 95 % cells equal under `rtol=1e-3`, not byte-identical | "passes at V2 tolerance; regression only under strict byte compare" |
| `rds_semantic_gap` | `.rds` expected, file present, sidecar could extract a tabular view but schema/content mismatches | "sidecar could not reconstruct S4 slot: <class name>" |
| `infinite_debug_loop` | ≥ 8 total steps AND ≥ 7 write/rscript attempts after the first write AND no file ever reaches `ok` | "agent retried K times after first write; consider raising max_steps or explicit recipe" |
| `mixed` | multiple distinct per-file modes in one task | "multiple failure modes across expected files — see per-file diff" |
| `task_never_started` | no `metadata.json` on disk (batch runner skipped the row — e.g. upstream OpenRouter 402) | "task never started (no metadata.json; batch runner skipped this row)" |

## Confidence grade

| grade | condition |
|-------|-----------|
| `high` | at least one expected file is `byte_identical`, OR a tabular file has ≥ 90 % cells matched with aligned schema |
| `medium` | otherwise, the best per-file `strategy_score` lies in `[0.5, 0.9]` |
| `low` | sidecar fallback was needed, OR best strategy_score < 0.5, OR an expected file was missing |

## Skill-attribution hint

V3 also loads a `skill_tokens_v3.json` file of the form
`{ task_id → { arm → [tokens] } }`. It scans the trajectory once for
each task's tool-call argument strings and records which of the arm's
skill tokens show up at least once. The resulting
`skill_tokens_matched` list answers: *did the agent actually surface
any skill-unique content in its actions?* Generated by running:

```bash
python3 tools/evaluators/skill_tokens.py
```

(which reads the three live manifests and emits a fresh
`evaluators/skill_tokens_v3.json`).

The token extractor (pure heuristic, deterministic):

* function-like identifiers followed immediately by `(`, e.g.
  `DESeqDataSetFromMatrix(`, `methRead(`,
* backtick-quoted code spans,
* `library(pkg)` and `pkg::fn` pairs,
* comparison thresholds like `padj < 0.05`, `log2FoldChange >= 2`.

The filter drops common English + generic R words (see the `_STOP` set
in `skill_tokens.py`). Paper-arm skills, which are vision-adapter
prose-summaries, therefore typically produce zero tokens — itself a
useful insight ("the paper arm provides no code-actionable signal to
look for in the trajectory").

## CLI

```
python3 tools/evaluate_real_run_v3.py \
    --batch-run-id sweep_v3_none_20260416T194356Z \
    --batch-run-id sweep_v3_paper_20260416T194356Z \
    --batch-run-id sweep_v3_pipeline_20260416T194356Z \
    --batch-run-id sweep_v3_llm_plan_20260416T194356Z \
    --skill-tokens-json tools/evaluators/skill_tokens_v3.json \
    --insight-only
```

`--insight-only` reads the cached `<batch>.v2.json` and avoids
recomputing V2 scores (fast, seconds). Drop it to run V2 + V3
end-to-end.

Outputs:

* `runs/_evaluations/<batch>.v3.json` — extends the V2 JSON with
  `insights`, `failure_mode_counts`, `confidence_counts`.
* `runs/_evaluations/<batch>.v3.md` — table with task × verdict × mode
  × confidence, plus a per-task "Why" section that prints the
  actionable fix, matched skill tokens, and per-file diff notes.

## Design constraints

* V3 never modifies V2 / V1 code paths.
* No new third-party dependencies beyond what V2 already uses.
* No LLM judges, no network access.
* All failure-mode thresholds are encoded as named constants at the top
  of `evaluators/insight.py` for easy auditing.
